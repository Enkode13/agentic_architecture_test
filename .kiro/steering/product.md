# Product Description & Project Goal

## Core Objective
Improve Generation 3 (`lang_core_multi/`) depending on the prompts provided.

## Domain Context
- **Dataset / Target Domain:** Advanced Q&A over historical quantum physics academic papers (specifically covering Heisenberg, Schrödinger, Einstein, and Planck). 
- **Expected System Output:** Highly precise, context-aware answers pulled from a specialized multi-agent pipeline using Groq inference.

## Project Evolution Context
The workspace preserves a 3-generation chronological architecture for experimental comparison. Do not modify Generations 1 or 2.
1. **Generation 1 (`core/`):** Manual ReAct Loop via raw Python loops (Legacy - **DO NOT ALTER**).
2. **Generation 2 (`lang_core/`):** Single-Agent LangGraph setup (Legacy - **DO NOT ALTER**).
3. **Generation 3 (`lang_core_multi/`):** Multi-Agent Orchestration layout (**CURRENT WORKSPACE**).

## Orchestration & Routing Logic (The 4 Routing Paths)
The Main Controller LLM evaluates the user input and updates the state to trigger one of four distinct routing behaviors:

1. **Only Tool Agent:** Routes to the Tool Agent sub-graph ➔ passes to `consolidation_node` ➔ returns to Main Controller for final synthesis.
2. **Only Retrieval Agent:** Routes to the Retrieval Agent sub-graph ➔ passes to `consolidation_node` ➔ returns to Main Controller for final synthesis.
3. **Sequential Dependency:** Routes to the Retrieval Agent first. If the output generated by the Retrieval Agent requires calculation or dictionary lookup, the graph routes its context into the Tool Agent sub-graph next, before moving to consolidation.
4. **Simultaneous Fan-Out (Parallel):** Triggered when a question requires both domain knowledge and tool execution. The graph executes both the Retrieval Agent and Tool Agent sub-graphs in parallel. The `consolidation_node` acts as a barrier synchronization point (Fan-In), waiting for both streams to finish and merging their outputs into a structured payload for the Main Controller.
